# 审查流程

## 1. 审查目标

每次改动、阶段性整理或发布前，审查不是只看“能不能跑”，而是同时确认：

- 改动是否符合 `[[01 PROJECT_RULES]]`
- 当前文档描述是否仍符合真实实现
- 本次改动是否真的接入主路径，而不是只增加了未生效代码
- 发布物是否通过基础构建检查

## 2. 审查顺序

1. 先读 `[[00 项目总览]]`、`[[01 PROJECT_RULES]]`、`[[04 接续说明]]`。
2. 用管理索引确认这次改动属于哪个边界，哪些文档可能需要同步更新。
3. 用代码索引检查目标功能的入口、调用链、依赖和影响面。
4. 对照代码实际状态，判断 `[[04 接续说明]]` 与其他项目文档是否需要修订。
5. 最后再执行构建类检查。

## 3. 双索引分工

- **管理索引负责**
  - 查规则
  - 查接续上下文
  - 判断该补哪些文档
  - 做文档一致性检查
- **代码索引负责**
  - 查真实实现
  - 查调用链
  - 查未接线路径
  - 查改动影响面

如果文档说“已实现”，但代码索引找不到主路径接线，按未完成处理；如果代码里存在函数，但没有进入主渲染或主命令路径，按未生效处理。

## 4. 发布前最低检查项

- `lint` 成功
- `build` 成功
- 本次改动涉及的真实入口和影响面已通过代码索引核对
- 本次改动涉及的项目文档已按需要同步更新

## 5. 与 GitHub 同步对齐

凡进入 GitHub 同步流程，审查流程默认视为已经启动，不允许把“发布检查”和“同步动作”拆成两套口径。

- **同步前强制动作**
  - 先刷新管理索引
  - 再刷新代码索引
  - 确认当前审查结论仍与最新代码和最新文档一致
- **同步前若仍有补写**
  - 只要在同步前继续补了 `03 改动日志`、`04 接续说明`、`README.md`、`README_en.md`、版本号或收尾代码，就必须在提交前再刷新一次最终索引
- **审查完成判定**
  - 只有当双索引、文档口径、代码实现、`lint`、`build` 五者一致时，才允许进入最终提交与推送

如果双索引与文档、代码任意一方不一致，优先修正不一致，再谈同步。
