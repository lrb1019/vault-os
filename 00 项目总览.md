# Vault OS 项目总览

## 项目定位

Vault OS 是一个 Obsidian 插件项目，目标是在 Obsidian 内提供一个统一的个人控制面板，用来汇总知识库状态、周期笔记、巡检结果、TickTick 数据和项目视图，并为智能体工作流提供统一入口。

当前阶段：插件主视图、设置面板和多项真实功能已经落地，当前重点不再是“准备做 UI 和真实功能”，而是继续对齐文档、收紧实现边界、稳定集成链路并逐步打磨交互与数据口径。

## 目录结构与协作要求

- **项目代码与管理目录**：`D:\OneDrive\Sync\.obsidian\plugins\vault-os`
  **要求**：所有插件代码的修改与项目管理文档（如本文件）均存放于此目录，实现文档与代码库的统一管理。

## 统一索引结构

- **项目与代码索引**
  - 索引对象：`D:\OneDrive\Sync\.obsidian\plugins\vault-os`
  - 用途：读取项目定位、规则、接续说明、审查流程、日志与备忘录，以及插件的真实代码实现、入口函数、调用链。工作时先查阅文档确定规则与上下文，再进行代码改动，改完后同步更新日志与文档。

## 开荒期原则

- 以长期最优机制为准。
- 优先验证最小可用闭环。
- 不为了兼容早期试验牺牲后续体验。
- 遵循 Obsidian 官方插件开发规范。

## 当前实现概况

- 插件已提供独立 View、Ribbon 按钮、命令入口和设置面板。
- 主界面当前以顶部状态栏 + 五个主频道的单页切换为主：
  - `01 / 仓库`
  - `02 / 日记`
  - `03 / 巡检`
  - `04 / TickTick`
  - `05 / 项目`
- 核心能力已经覆盖：
  - 知识库统计与热力图
  - 周期笔记创建与摘要预览
  - 巡检与体检报告入口
  - TickTick 缓存 + MCP 同步
  - `Projects.base` 驱动的项目表格视图
- 现阶段文档必须围绕“已实现了什么、哪些入口已生效、哪些代码仍处于未接线状态”来写，不能再按最初设想描述。

## 文档分工 (核心 8 文件体系)

为保持项目管理文档的纯净与规范，项目当前保留以下 8 个核心文档：

### 1. 想法：[[02 后续想法]]
- 记录未来需求、灵感、反思。
- 勾选已完成的任务，整理分类想法（作为项目的 Backlog）。

### 2. 日志：[[03 改动日志]]
- 每次改动的记录。
- 干了什么、沉淀了哪些技术经验记录。

### 3. 接续说明：[[04 接续说明]]
- 对于现状的整体介绍，代码整体结构。
- 分别对应什么功能，相应功能是怎么实现的。
- 有什么避坑的、或者比较成功的方式提醒（包含历史改造复盘）。

### 4. 审查：[[05 审查流程]]
- 每次改动、阶段性结构审查和发布前的检查流程与清单。

### 5. 项目总览：[[00 项目总览]]
- 项目定位、原则与整体结构介绍。

### 6. GitHub 同步：[[06 GITHUB_SYNC]]
- 专属的 GitHub 快捷发布指南。
- 每次用户说“同步到 GitHub”时走这个捷径流程。

### 7. 规则：[[01 PROJECT_RULES]]
- 具体功能或整体的硬性规则要求，必须遵守的边界。
- 包含 GitHub 同步要求、插件数据架构边界等。

### 8. 审计与体检逻辑：[[07 审计与体检逻辑备忘录]]
- 知识库健康审计、体检指标的底层判断依据、数据来源以及代码实现方式。
- 包含死链、孤儿笔记的底层逻辑判定。
- 后续凡涉及“指标抓取算法”、“统计口径”或“评判标准”的更新，必须同步更新该备忘录。

## 以后怎么用

每次开始改 Vault OS 前，先读：

1. [[00 项目总览]] (本文件)
2. [[01 PROJECT_RULES]]
3. [[04 接续说明]]

读完文档后，必须继续做两步：

4. 用管理索引确认这次改动属于哪个边界，应该同步哪些文档。
5. 用代码索引确认本次目标功能的入口、调用链、依赖与影响面。

如果是新想法：写进 [[02 后续想法]]。
如果是一次重要改动：改完后记录到 [[03 改动日志]]。
如果需要同步到 GitHub：严格遵守 [[01 PROJECT_RULES]] 中的同步发布规则。
