# Vault OS

Vault OS 是一个 Obsidian 个人知识工作首页。它把当天记录、周期复盘、知识库体检、个人智能指令与每日阅读回看集中到一个自定义视图中。

## 当前范围

当前一级入口为：

- **首页**：每日一句、今日日记、周期复盘、体检入口和每日阅读回看。
- **周期复盘**：日、周、月、季、年笔记的打开、创建和去年同期回望。
- **知识库体检**：工作流诊断、Inbox、待入库日记、孤儿笔记、死链和空白笔记。
- **智能指令**：按用户自定义分类组织的 Skill 触发面板。

Vault OS 不复制任务管理器或项目管理器。时间管理和执行留在专用工具中；Vault OS 专注于知识沉淀、记录、复盘和维护。

## 安装

1. 下载 Release 中的 `main.js`、`manifest.json`、`styles.css`。
2. 放入 `你的仓库/.obsidian/plugins/vault-os/`。
3. 重新加载 Obsidian 后，在社区插件中启用 **Vault OS**。

## 初始设置

首次启用后，按自己的仓库情况选择：

1. 空白仓库可以直接使用首页、空白笔记扫描和周期笔记入口；未配置的能力不会显示伪造数据。
2. 已有仓库可在 `仓库规则` 中扫描 Inbox 候选。候选只分析路径、标签和 frontmatter，必须由你确认后才保存。
3. 需要周期笔记模板时，配置 `Notebook Navigator` 与 `Templater`；未安装时 Vault OS 使用手动周期文件名回退规则。
4. 需要智能指令时，安装并启用 `realclaudian`，然后在设置页维护自己的动作。

## 仓库规则

Vault OS 的核心不依赖 `01 Daily`、`02 Inbox` 等固定目录。它以语义角色识别仓库内容：周期笔记、Inbox、知识范围、Projects 和 Output。

当前 `仓库规则` 支持为 Inbox 选择：

- 一个文件夹，可选择是否包含子文件夹。
- 一个或多个标签。
- frontmatter 属性和值。

首次使用只需在“快速开始”中确认全局安全排除，并按需要点击“应用当前仓库推荐映射”。它会启用现有仓库的 Project、Question、Claim、Evidence 与 Output 基础识别，但不会定义 P0 优先级或 Output 的发布/复盘状态。其余规则默认收在“高级规则”，可在工作流稳定后再调整。
- 整个 Markdown 仓库。

旧路径设置仍会在运行时生成兼容规则，因此现有仓库不会因升级而改变 Inbox 的直接子文件扫描范围。

工作流诊断额外使用以下可选规则：

- **全局安全排除文件夹**：必须至少配置一项。未配置时语义巡检安全停止，不读取 frontmatter、链接或正文；排除范围也不会进入快照、报告或问题列表。
- **Project 实体入口**：Project 的识别优先级为 `layer: project`、本规则、文件夹同名主页。Projects 范围内的其他 Markdown 不会被误判为项目。
- **Project 状态兼容映射**：将显示状态映射为 `pending`、`active`、`on-hold`、`blocked`、`completed`、`cancelled` 或 `unknown`。
- **Output 实体入口与生命周期**：只有同时配置 Output 实体入口、已发布状态和已复盘状态时，才会提示“已发布但未复盘”；否则显示“不可判定”。
- **知识实体契约**：分别定义 Question、Claim、Evidence 的识别规则。关系口径固定为 Question 与 Claim 的双链、Output 指向 Claim 的出链，以及 Evidence 的 `supports`；没有完整契约时不扫描知识图谱。
- **P0 Claim 识别规则**：只有同时满足 Claim 类型和该规则的笔记才会进入证据债务诊断；没有明确 P0 标记时保持“不可判定”。

## 周期复盘

周期页支持日、周、月、季、年五种周期：

- `Notebook Navigator` 可用时，读取其周期目录、命名规则和模板路径。
- 没有该插件时，手动模式生成不同的默认目标：`YYYY-MM-DD`、`YYYY-Www`、`YYYY-MM`、`YYYY-Qn`、`YYYY`。
- 创建前由用户显式点击；创建失败会显示原因。
- 页面提供当前周期和去年同期入口，但不会替你自动写复盘结论。

## 知识库体检

通用指标：

- 未解析死链。
- 空白 Markdown 文件。

需要配置范围的指标：

- Inbox 积压。
- 待入库日记。
- 孤儿笔记。

工作流诊断按知识链路只读检查以下问题：

- `completed` 但仍位于 Projects 范围内的 Project 实体。
- 已发布但尚未复盘的 Output 实体（仅在生命周期映射完整时判断）。
- 被真实 Project 实体或 Output 实体链接、但没有 `Evidence.supports` 结构化证据的活跃 Claim。
- 未声明 `supports`、`supports` 格式无效，或 `supports` 指向不存在目标的 Evidence。
- 尚未链接 Claim 的 Question 与 Output 候选。普通双链只证明关联，不会被断言为“已回答”或“已完成”。
- 满足用户定义 P0 规则、但没有 `Evidence.supports` 结构化证据的 Claim；未配置 P0 规则时保持不可判定。
- 与用户主动保存的上一份快照相比，新增、持续和已解决的问题。

最小知识链为：`Question` 通过双链关联 `Claim`；`Evidence` 使用 `supports: "[[Claim]]"` 或字符串数组声明支持的判断；`Output` 出链链接其使用的 `Claim`。不需要改造旧笔记，可从新建的研究、项目或输出逐步采用。

点击结果可查看路径及其确定性触发依据，再打开对应笔记。点击“保存当前基线”才会写入快照。快照只保留问题标识、标题和时间，不保存 Markdown 正文，不会触发修复；诊断范围或状态规则变化后会自动失去可比性。Vault OS 提供判断依据和动作入口；Skill 负责补证据、修复和知识回流。

空白笔记回收遵循固定流程：扫描候选 → 勾选范围 → 确认声明 → 移动到 Obsidian 回收站 → 显示逐项结果。归档同样会预览来源和目标，并拒绝同名覆盖。

月度巡检报告写入设置中的输出目录；同名报告不会被覆盖。

## 智能指令

智能指令完全由设置页定义，不内置固定 Skill 清单。它有独立一级入口，不再混在知识库体检中。每条指令可配置名称、图标、提示词、是否需要输入、占位说明和所属分类；分类本身也可新增、编辑、删除。删除分类时，其中的指令会回到“未分类”，不会丢失。

模板变量：

- `{{input}}`
- `{{daily_path}}`
- `{{inbox_path}}`
- `{{atomics_path}}`
- `{{archive_path}}`
- `{{output_path}}`

Vault OS 只负责变量替换、Claudian 可用性检查和调用结果。依赖 Claudian 私有窗口与 DOM 的部分被隔离；插件不可用时会明确提示而不会丢失你的配置。

## 开发

需要 Node.js 18 或更高版本。

```text
npm install
npm run verify
```

`npm run verify` 依次运行：

1. Node 自动测试。
2. TypeScript 类型检查和生产构建。
3. ESLint。
4. 生成的 `main.js` 语法检查。

源码位于 `src/`，根目录 `main.js` 由构建生成，不应手工编辑。自动测试不能替代 Obsidian 隔离测试 Vault 的实机验收。
