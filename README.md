# Vault OS

一个 Obsidian 插件，用于在一个自定义视图中统一呈现知识库活动、周期笔记、知识库巡检、TickTick 数据和项目追踪。

## 当前范围

插件当前已经提供：

- 自定义 dashboard 视图
- Ribbon 快捷入口
- `Open Vault OS` 命令
- 设置面板，用于配置路径、TickTick/MCP、热力图尺寸和自定义动作

当前实际生效的界面，是一个顶部状态栏加五个主频道：

- `01 / 仓库`
- `02 / 日记`
- `03 / 巡检`
- `04 / TickTick`
- `05 / 项目`

## 各频道当前实际功能

- **仓库**：统计知识库分类数据，渲染迷你指标卡，并支持柱状图、日历图和热力图切换。
- **日记**：读取 `notebook-navigator` 的周期笔记规则；在可用时通过 `templater-obsidian` 创建笔记，并展示当前周期与去年同期预览。
- **巡检**：扫描收件箱积压、未入库日记、孤儿笔记、死链和空白笔记，同时提供可配置的 Claudian 动作按钮。
- **TickTick**：优先加载本地缓存，再通过 `mcp.json` 中定义的 HTTP MCP 端点同步任务、习惯、专注和项目数据。
- **项目**：读取 `Projects.base`，解析其中的过滤条件，并渲染项目表格，展示状态、创建时间和 topics。

## 重要说明

- 代码库中仍然保留了一些未挂载到当前界面的辅助渲染函数，所以不是所有现存函数都属于当前在线界面。
- 当前 `项目` 频道**不是** Kanban，看的是 `Base` 驱动的表格视图。
- TickTick 功能依赖可用的 MCP 配置；如果 MCP 不可用，界面会回退到占位数据。

## 安装指南

### 手动安装

1. 下载最新发布的 Release 资源文件 (`main.js`、`manifest.json`、`styles.css`)。
2. 将这些文件复制到你金库的插件目录中：`你的金库路径/.obsidian/plugins/vault-os/`。
3. 重新加载 Obsidian，并在社区插件设置中启用 **Vault OS**。

## 使用说明

### 先做什么

第一次启用后，建议先打开插件设置并完成这四类配置：

1. 填写 `知识库路径`，先把日记、Inbox、原子笔记、项目、归档、输出这些目录指向你自己的实际文件夹。
2. 如果你要使用日记自动创建，请确认是否要配合 `notebook-navigator` 与 `templater-obsidian`。
3. 如果你要使用巡检下方的自定义按钮，请先安装并启用 `realclaudian`。
4. 如果你要使用 TickTick，请在 `TickTick 连接` 中填好接口地址和请求头。

### 设置页各标签控制什么

- `看板基础设置`
  - 当前只保留看板主标题。
  - 低频技术参数已经从界面中移除，避免设置继续失控膨胀。
- `知识库路径`
  - 这些路径不是装饰项，而是在定义整个插件的数据口径。
  - 日记文件夹里的全部内容都会被视为日记来源。
  - 原子笔记文件夹决定巡检里的核心知识范围。
  - 项目文件夹与 Base 文件共同决定项目页的数据源。
- `TickTick 连接`
  - TickTick 连接现已由本插件独立管理，不再依赖其他插件共用配置文件。
  - 当前真正会影响连接结果的字段只有：启用状态、接口地址、请求头、服务标识符、高级同步延迟。
- `智能指令`
  - 这里配置的是巡检页底部的自定义 Skill 按钮。
  - 它们本质上不是“插件内部功能”，而是把指令模板转交给 `realclaudian` 执行。

### 01 / 仓库

用途：

- 统计整个仓库的 Markdown 文件数量、记录天数、日均新增、文件夹分类占比，以及按日期汇总的图表。

你需要做什么：

- 先把 `日记 / Inbox / 项目 / 原子笔记 / 输出 / 归档` 路径设置好。
- 这些路径会直接影响仓库页里“Daily / Projects / Other”这类分类统计。

它怎么判断数据：

- 仓库概览按整个金库扫描 Markdown 文件。
- 记录天数按仓库中最早文件的真实创建时间计算。
- 图表日期优先使用日记文件名日期或 frontmatter 的 `created`，否则回退到文件创建时间。

### 02 / 日记

用途：

- 显示今日日记状态、统计周期笔记数量，并提供日 / 周 / 月 / 季 / 年笔记的创建入口与去年同期预览。

你需要做什么：

- 在设置中指定一个“日记文件夹路径”。
- 只要文件位于这个文件夹中，它就会被当作日记来源参与统计。

推荐配合：

- `notebook-navigator`
  - 如果已安装，插件会优先读取它当前激活 profile 的周期命名规则、根目录和模板路径。
- `templater-obsidian`
  - 如果已安装，并且 `notebook-navigator` 配置了模板，插件会优先按模板创建周期笔记。

回退行为：

- 如果没有上述插件，插件会在你设置的日记目录中直接创建默认 Markdown 文件。
- 默认模板会写入：
  - `created: YYYY-MM-DD`
  - `author: "[[Jarvis]]"`
  - `ingested: false`

### 03 / 巡检

用途：

- 用统一面板检查收件箱积压、待入库日记、孤儿笔记、死链和空白笔记，并给出健康度。

你需要做什么：

- 配好 `Inbox`、`日记文件夹`、`原子笔记文件夹`、`输出文件夹`。
- 如果要使用下方自定义按钮，请安装并启用 `realclaudian`。

当前巡检逻辑：

- `待分类文件`
  - 只统计 `Inbox` 文件夹中的 Markdown 文件。
- `待入库日记`
  - 只统计日记文件夹中的 Markdown 文件，且 frontmatter 中 `ingested !== true`。
- `孤儿笔记`
  - 默认只检查原子笔记文件夹中的笔记。
  - 链接来源会参考原子笔记、输出、日记三类内容。
  - 会排除 `Index` 文件与体检报告，避免误判。
- `未解析死链`
  - 只统计来源于原子笔记、输出、Inbox 的未解析链接。
- `空白笔记`
  - 当前按全库扫描。
  - 如果一个文件去掉 frontmatter 和纯标题行后没有任何正文内容，就会被视为空白笔记。
  - 这意味着“完全空文件”“只有标题”“只有属性没有正文”都会被识别出来。

#### Claudian 自定义按钮怎么用

这些按钮位于巡检页底部。

- 无输入框按钮
  - 点击后，插件会把预设指令直接发送给 `realclaudian`。
- 带输入框按钮
  - 如果设置中开启了“独立输入框”，按钮左侧会出现一个输入框。
  - 你可以在指令模板里使用 `{{input}}`，运行时会自动替换为这次输入的内容。
- 路径变量
  - 指令模板中支持 `{{daily_path}}`、`{{inbox_path}}`、`{{projects_path}}`、`{{archive_path}}`、`{{output_path}}`、`{{atomics_path}}`。
  - 这些变量会自动替换为你在设置中填写的真实路径。
- 图标
  - 图标名称来自 [Lucide Icons](https://lucide.dev/icons/)。
  - 把网站上的图标名填进设置页即可。

### 04 / TickTick

用途：

- 展示今日任务、完成情况、习惯、专注统计，并通过 MCP 连接 TickTick。

你需要做什么：

- 在 `TickTick 连接` 中填好：
  - 启用状态
  - 接口地址，例如 `https://mcp.ticktick.com`
  - 请求头，例如 `Authorization=Bearer ...`
- 一般不需要改服务标识符，默认保持 `ticktick` 即可。

当前运行方式：

- 进入频道时，插件会优先读取本地缓存，让界面先显示出来。
- 随后再尝试同步远端 TickTick 数据。
- 如果连接不可用，界面会回退到提示性占位数据。

### 05 / 项目

用途：

- 用一个 Base 文件定义“什么样的笔记算项目”，再把这些项目笔记汇总成统计卡片和表格。

先理解核心机制：

- 项目页的数据源不是“项目文件夹里的所有文件直接全上”。
- 它的真实逻辑是：
  - 先读取你设置的 `Projects.base`
  - 再用 `Projects.base` 里的 `filters` 去匹配全库 Markdown 文件
  - 只有匹配到过滤条件的笔记，才会进入项目页

所以你需要同时准备两样东西：

1. 一个项目文件夹，用来存放你的项目 Markdown 笔记。
2. 一个 `Projects.base` 文件，用来声明筛选规则。

推荐项目笔记模板：

```md
---
status: active
topics:
  - "[[AI]]"
  - "[[Obsidian]]"
progress: 35
deadline: 2026-07-15
tags:
  - project
---

# 项目标题

> [!GOAL] 核心目标
> 用一句话说明这个项目最核心的目标。
```

推荐说明：

- `status`
  - 推荐使用：`pending`、`active`、`on hold`、`blocked`、`completed`、`cancelled`
  - 也兼容你已有的彩色状态写法，例如 `🟢`、`🟡`、`🔴`、`🔵`、`⚫`
- `topics`
  - 支持字符串数组。
  - 支持 `[[双链]]`，项目页里会渲染成可点击标签。
- `progress`
  - 当前主要作为项目元数据保留，便于后续扩展。
- `deadline`
  - 当前会被读取，但表格主列暂不直接展示。

推荐 `Projects.base` 过滤示例：

```yaml
filters:
  and:
    - file.path.contains("03 Projects/")
    - file.tags.contains("project")
```

当前表格展示什么：

- 名称：来自文件名
- 状态：来自 frontmatter 的 `status`
- 创建时间：来自文件真实创建时间
- topics：来自 frontmatter 的 `topics`

如果你看到的是示例项目，而不是你自己的真实项目，通常说明：

- `Projects.base` 没匹配到任何文件
- 项目笔记没有满足 `filters`
- 你只是建了项目文件夹，但没有把笔记结构和 Base 规则配起来

## 开发与编译

确保本地安装了 NodeJS >= v18。

1. 克隆本仓库。
2. 安装依赖：
   ```bash
   npm install
   ```
3. 启动开发 watch 监听服务：
   ```bash
   npm run dev
   ```
4. 运行生产环境编译打包：
   ```bash
   npm run build
   ```
5. 运行代码 Lint 静态检查：
   ```bash
   npm run lint
   ```
