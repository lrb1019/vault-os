- 2026-06-27: 补充记录“仓库页顶部周期切换区首次加载时从图一过渡到图二”的专项排查与修复总结：
  1. 现象：进入 `01 / 仓库` 首屏时，左侧“周 / 月 / 年 / 全部”切换区与右侧时间区间虽然不涉及复杂计算，但首帧仍会先显示一版初始样式，随后再跳到最终样式；此前最明显的表现是按钮挤压、容器气质变化和日期区字号失衡，后续收敛后仍保留了一个“先绿后红”的状态切换。
  2. 根因：问题核心不在“数据慢”，而在“首帧 DOM 结构和最终视觉状态不是同一套输出”。一方面，`renderVaultDashboard` 原先先画骨架，再通过 `setTimeout(..., 0)` 回填正式内容；另一方面，顶部周期切换区和日期区一开始主要依赖外部 class 样式接管，导致浏览器先用初始布局渲染，再被后续样式覆盖，所以用户看到的是 UI 变化，而不是单纯数字变化。
  3. 本次解决办法：将顶部周期切换区、日期外框、左右箭头和日期文本的关键视觉参数在创建 DOM 时就直接写成接近最终态的样式；同时让仓库页首轮渲染优先直接走最终布局路径，减少“先骨架、后替换”的视觉差；另外补充 `scrollbar-gutter: stable`，降低首屏滚动条占位变化对横向排版的二次扰动。结果是布局跳变基本被消掉，只保留了可接受的颜色态变化，这个变化也能顺手承担“正在同步 / 正在进入稳定态”的提示作用。
  4. 复盘：之前围绕“日期字号”“缓存”“扫描速度”“数据计算”做的修补，只能改善表层症状，不能根治这个问题；真正的根因是首帧和终帧不是同一个 UI 输出，所以真正有效的修法是锁定首帧视觉，而不是继续优化后端扫描。
  5. 如果之后要一步到位彻底改干净，建议这样做：第一，直接统一顶部周期按钮的初始激活态，让它首帧就等于最终红色态，不再经历绿色过渡；第二，继续收缩 `renderStatsNav` 的状态分支，保证首次执行时只产生一种视觉结果；第三，如仍有残余变化，再往外层排查 Obsidian 视图容器二次布局、主题字体接管时机和首帧宽度重算，而不是再把注意力放回数据计算。换句话说，后续如果要彻底“秒切图二”，方向应该是减少首屏 UI 状态数，而不是继续压榨扫描性能。
